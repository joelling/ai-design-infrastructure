# Chapter 5: Information Architecture

> **Tier 2 — Definition** | Mode: `design-ia`

## Why this matters

IA is where abstract user needs become concrete structure. It answers: what screens does this product need, how do users navigate between them, and how is content organized within each screen? This is the **direct bridge to Figma** — the screen inventory becomes the Figma page list, the content hierarchy becomes the component checklist.

This is also the first mode where technology decisions intentionally start appearing. IA bridges the agnostic upstream with the concrete downstream.

## The mental model

Think of yourself as an architect drawing the floor plan before any interior design begins. You're deciding: how many rooms, what's each room for, how do people move between rooms, and what goes in each room (at a category level, not furniture level).

The key question for every screen: "Why does this screen exist, and which personas need it for which stories?"

## Inputs

- `design/journeys/task-flows/*` — task flows reveal what screens are needed
- `design/stories/story-map.md` — stories define what each screen must support
- `design/user-models/personas/*` — different roles may need different navigation
- Spec data model — defines what information exists to display

## Process

**1. Build the screen inventory.** From task flows and story maps, identify every distinct screen or view. For each, document: screen name, purpose, primary persona(s), stories served, and entry points. Distinguish between full screens (own route), views within screens (tabs, panels), and overlays (modals, drawers).

**2. Design the navigation model.** Define global navigation (always visible), local navigation (contextual), role-based navigation variations, and entry points. Consider: can the product be entered from multiple places?

**3. Map content hierarchy.** For each screen, define what information appears in three levels: primary (immediately visible), secondary (visible on scroll or interaction), and tertiary (available on demand). List available actions and who can perform them.

**4. Define the taxonomy.** How is content categorized and labeled across the product? Content types, categorization scheme, label conventions.

## Outputs

| File | What it contains |
|------|-----------------|
| `design/information-architecture/sitemap.md` | Complete screen inventory with purpose, personas, stories |
| `design/information-architecture/navigation-model.md` | Global nav, role-based variations, entry points |
| `design/information-architecture/content-inventory.md` | Per-screen content hierarchy |
| `design/information-architecture/taxonomy.md` | Content categorization and label conventions |

## Rules

- Every screen must trace back to at least one user story or task flow. No orphan screens.
- Navigation must account for all roles — document variations explicitly.
- Content hierarchy uses three levels: primary, secondary, tertiary. Don't flatten everything.
- Screen names established here become canonical — used everywhere downstream in Figma pages, canvas briefs, and component names. Choose them carefully.
